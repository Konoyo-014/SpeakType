# v2.1.1 Latency Design

## Goal

SpeakType v2.1.1 focuses on end-to-end responsiveness without changing the product contract. Audio stays local, ASR stays on Qwen3-ASR, and the final inserted text should preserve the same quality expectations as v2.1.0. The design target is simple: move unavoidable work earlier, overlap independent stages, and remove non-user-visible tail latency from the hot path.

## Strategy

The first optimization layer treats recording time as usable compute budget. If the ASR model is still cold when dictation starts, the app now begins loading it immediately in the background instead of waiting until the user releases the hotkey. The same idea applies to Ollama. When polish or translation is enabled, the app now sends a background prewarm request and keeps the model alive for a bounded window so the first real chat request is less likely to pay the full cold-start cost.

The second layer reduces preview latency. Streaming preview no longer depends entirely on the ASR engine having finished loading before recording starts. When recording begins early in app startup, the app now watches for the ASR engine to become ready and starts streaming preview as soon as that becomes possible. The preview loop itself also starts with a smaller audio buffer and a shorter refresh interval, which lowers the time to first partial text without changing the final transcript path.

## Hot Path Changes

The dictation hot path now records phase timings with `time.perf_counter()` so future optimization work has concrete numbers in logs instead of guesswork. These timings cover transcription, polish, translation, insertion, and history enqueue. They are intentionally diagnostic only and do not change user-facing behavior.

The post-insert history write has been removed from the synchronous user-facing path. History persistence now runs in a background thread behind a lock, which means the user sees completion as soon as insertion is done instead of waiting on JSON serialization and fsync-heavy file writes.

## Validation

The implementation is validated with focused tests for asynchronous ASR loading reuse, Ollama keep-alive and prewarm requests, recording-time pipeline priming, and asynchronous history persistence. The full suite remains green after the change set. This design is intentionally conservative about semantics: it optimizes when work starts and when non-critical work finishes, but it does not change the final text generation rules.
